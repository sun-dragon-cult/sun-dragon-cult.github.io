## RQID lookup now uses global priority across World and Compendiums

<GithubIssue issue="829" repo="fvtt-system-rqg" />

This release changes how RQID matches are selected when both World documents and Compendium
documents exist.

The main change is that RQID lookup now picks the highest priority match globally, instead of always
preferring World first. If two matches have the same priority, World still wins the tie-break.

#### What changed

- Rqid.fromRqid now compares World and Compendium matches by priority and returns the highest
  priority match overall.
- Rqid.fromRqidRegexAll has been replaced by Rqid.fromRqid with options for source and mode.
- Rqid.fromRqidCount now takes an options object instead of a scope string.
- Rqid.getDocumentType now returns undefined when the rqid type segment is empty (for example
  je..my-journal).

Why this was changed The previous behavior could return a lower-priority World match even when a
higher-priority Compendium match existed. With this update, priority behaves consistently across all
RQID lookup paths.

Breaking API changes for macros and integrations If you have custom macros/modules using the old
method signatures, update them as follows:

#### Old -> New syntax

`game.system.api.rqid.fromRqidRegexAll(regex, docPrefix, lang, world)`
`game.system.api.rqid.fromRqidRegex(regex, docPrefix, lang, { source: world, mode: all })`

`game.system.api.rqid.fromRqidRegexAll(regex, docPrefix, lang, packs)`
`game.system.api.rqid.fromRqidRegex(regex, docPrefix, lang, { source: packs, mode: all })`

`game.system.api.rqid.fromRqidRegexAll(regex, docPrefix, lang, match)`
`game.system.api.rqid.fromRqidRegex(regex, docPrefix, lang, { source: all, mode: best })`

`game.system.api.rqid.fromRqidCount(rqid, lang, world)`
`game.system.api.rqid.fromRqidCount(rqid, lang, { source: world })`

`game.system.api.rqid.fromRqidCount(rqid, lang, all)`
`game.system.api.rqid.fromRqidCount(rqid, lang, { source: all })`

#### Notes

- RQID Editor duplicate checks have been updated to use the new source and mode options.
- Existing worlds do not need data migration for this change, but custom scripts/macros may need API
  call updates.

## Make Protection token effect use the new regex Rqids

<GithubIssue issue="836" repo="fvtt-system-rqg" />

The protection token effects now use the regex Rqids introduced in the previous release (5.3.0).

## Allow Active Effects to affect multiple items

<GithubIssue issue="571" repo="fvtt-system-rqg" />

It is now possible to create Active Effects that affect multiple items. By prepending the Rqid
string with a ~ the Rqid will behave as a regex and all items that match that regex will be affected
by the Active Effect.

An example of what this could be used for is creating an Active Effect that applies to all hit
location items like this: `~i.hit-location`. That would match any hit location like
`i.hit-location.head` or `i.hit-location.left-wing`. You can also use the normal regex syntax to
build something more complex like `~i.cult.*orlanth.*` to find any cult that contains the word
orlanth.

For people who know how to write regex: yes, the dots in, for example, `~i.hit-location` will be
interpreted as "any one character". But in practice that should not matter. You could write it as
`~i\.hit-location` to be more strict if you wanted to.

## Attacking from prone

<GithubIssue issue="789" repo="fvtt-system-rqg" />

There is now a new checkbox in the attack dialog where you can select that you are attacking from a
prone position. This follows the rulebook by changing the hit location die to 1d10 (From Below) and
removing the damage bonus if you are not attacking with a natural weapon.

![](prone-attacker.png)

## Color the RQID icon red if it's missing

<GithubIssue issue="813" repo="fvtt-system-rqg" />

The sheet header RQID button is now colored red if the document does not have an RQID. This makes it
easier to see at a glance if there is a problem.

![](rqid-missing-header-icon.png)

## RQID editor now handles embedded documents

<GithubIssue issue="742" repo="fvtt-system-rqg" />

The RQID editor previously did not handle embedded documents like pages in a journal. It now does,
that, making it possible to link to a specific page for example. It could also be used to find other
embedded documents like items in an actor, but I think the main use for this is pages in journals.

With this you could add links directly to pages in another journal using rqid. Copy the link from
the example at bottom of the editor and paste that into any other journal.

![](embedded-document-working.png)

It will warn if the parent document does not have an RQID set since the embedded RQID will not work
in that case, see below.

![](embedded-document-rqid.png)
